OpenAgno sigue la superficie de interfaces expuesta por Agno en el runtime actual: WhatsApp, Slack, Telegram, AG-UI y A2A. El canal `ai_sdk` no forma parte de la ruta compatible vigente y no debe usarse en nuevas configuraciones.

## Matriz rapida

| Canal o protocolo | Activacion | Requisitos |
|-------------------|------------|------------|
| WhatsApp Cloud API (un solo tenant) | `channels: [whatsapp]` + `whatsapp.mode: cloud_api` | variables `WHATSAPP_*` |
| WhatsApp Cloud API (multi-tenant via Cloud) | Onboarding de Cloud + fila en `whatsapp_cloud_channels` de Supabase | `CHANNEL_SECRETS_KEY` compartida con Cloud |
| WhatsApp QR | `channels: [whatsapp]` + `whatsapp.mode: qr_link` | bridge Node opcional |
| Slack | `channels: [slack]` | `SLACK_TOKEN`, `SLACK_SIGNING_SECRET` |
| Telegram | `channels: [telegram]` | `TELEGRAM_TOKEN` |
| AG-UI | `channels: [agui]` | extra `.[protocols]` |
| A2A | `a2a.enabled: true` | extra `.[protocols]` |

## WhatsApp Cloud API

OpenAgno soporta dos modos complementarios de Cloud API.

### Un solo tenant (self-hosting, gestionado por el operador)

Si auto-hospedas OpenAgno para una unica cuenta de operador, define las credenciales como variables de entorno. El runtime monta una sola interfaz Agno `Whatsapp` en `/whatsapp/webhook` que usa estos valores.

```bash
WHATSAPP_ACCESS_TOKEN=tu_token
WHATSAPP_PHONE_NUMBER_ID=tu_phone_id
WHATSAPP_VERIFY_TOKEN=tu_verify_token
WHATSAPP_APP_SECRET=tu_app_secret
```

Rutas:

- `GET /whatsapp/webhook`
- `POST /whatsapp/webhook`

Este es el modo que el wizard del CLI configura cuando eliges "Cloud API" en `openagno init`.

### Multi-tenant (OpenAgno Cloud)

Cuando el runtime esta desplegado detras de OpenAgnoCloud, cada tenant puede traer su propia cuenta Meta con su propio Access Token, Phone Number ID, Verify Token y App Secret opcional. Cloud guarda estas credenciales cifradas (AES-256-GCM) en la tabla `whatsapp_cloud_channels` de Supabase y el runtime OSS expone un webhook dedicado por tenant.

Requiere una clave de 32 bytes en base64 compartida entre Cloud y OSS:

```bash
CHANNEL_SECRETS_KEY=$(node -e "console.log(require('crypto').randomBytes(32).toString('base64'))")
```

Rutas (por tenant, donde `tenant_id` es el UUID de `tenants.id` en Cloud):

- `GET /whatsapp-cloud/{tenant_id}/webhook` — verificacion Meta
- `POST /whatsapp-cloud/{tenant_id}/webhook` — eventos entrantes

El cliente pega esta URL y el Verify Token generado en Meta Developer Console &gt; App &gt; WhatsApp &gt; Configuration &gt; Webhook, y se suscribe al campo `messages`. El runtime carga las credenciales del tenant, las descifra con `CHANNEL_SECRETS_KEY`, valida `X-Hub-Signature-256`, resuelve el agente del tenant via `TenantLoader`, ejecuta y responde con Graph API usando el Access Token del propio tenant.

Si `CHANNEL_SECRETS_KEY` no esta definida, el webhook multi-tenant no se monta y el runtime sigue sirviendo el `/whatsapp/webhook` de un solo tenant desde Agno.

## Desarrollo local sin dominio

Durante desarrollo local normalmente necesitas un tunel publico para webhooks de WhatsApp, Slack o Telegram. En OpenAgno, la ruta practica esta documentada en [/es/whatsapp-tunnel](/es/whatsapp-tunnel).

Para pruebas controladas sin firma real:

```bash
WHATSAPP_SKIP_SIGNATURE_VALIDATION=true
```

Usalo solo en desarrollo.

## WhatsApp QR

```yaml
whatsapp:
  mode: qr_link
  qr_link:
    bridge_url: http://localhost:3001
```

```bash
cd bridges/whatsapp-qr
npm install
node index.js
```

## Slack

```bash
SLACK_TOKEN=xoxb-tu-token
SLACK_SIGNING_SECRET=tu_signing_secret
```

## Telegram

```bash
TELEGRAM_TOKEN=tu_token_de_botfather
```

## AG-UI y A2A

```bash
pip install -e '.[protocols]'
openagno add agui
openagno add a2a
```

Notas de compatibilidad:

- `slack` y `telegram` forman parte de la instalacion base.
- `agui` y `a2a` requieren extras de protocolos.
- `ai_sdk` no forma parte del runtime compatible actual de OpenAgno.
