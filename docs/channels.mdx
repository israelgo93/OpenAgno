OpenAgno follows the interface surface currently exposed by Agno: WhatsApp, Slack, Telegram, AG-UI, and A2A. The `ai_sdk` channel is not part of the supported runtime path and should not be used in new configurations.

## Quick matrix

| Channel or protocol | Activation | Requirements |
|---------------------|------------|--------------|
| WhatsApp Cloud API (single-tenant) | `channels: [whatsapp]` + `whatsapp.mode: cloud_api` | `WHATSAPP_*` env vars |
| WhatsApp Cloud API (multi-tenant) | External system writes encrypted row in `whatsapp_cloud_channels` | `CHANNEL_SECRETS_KEY` shared between writer and runtime |
| WhatsApp QR | `channels: [whatsapp]` + `whatsapp.mode: qr_link` | Optional Node bridge |
| Slack | `channels: [slack]` | `SLACK_TOKEN`, `SLACK_SIGNING_SECRET` |
| Telegram | `channels: [telegram]` | `TELEGRAM_TOKEN` |
| AG-UI | `channels: [agui]` | extra `.[protocols]` |
| A2A | `a2a.enabled: true` | extra `.[protocols]` |

## WhatsApp Cloud API

OpenAgno supports two complementary WhatsApp Cloud API modes.

### Single-tenant (self-hosting, operator-managed)

If you self-host OpenAgno for a single operator account, set the credentials as environment variables. The runtime mounts a single Agno `Whatsapp` interface at `/whatsapp/webhook` that uses these values.

```bash
WHATSAPP_ACCESS_TOKEN=your_token
WHATSAPP_PHONE_NUMBER_ID=your_phone_id
WHATSAPP_VERIFY_TOKEN=your_verify_token
WHATSAPP_APP_SECRET=your_app_secret
```

Routes:

- `GET /whatsapp/webhook`
- `POST /whatsapp/webhook`

This is what the CLI wizard configures when you answer "Cloud API" in `openagno init`.

### Multi-tenant

When you run this runtime behind an external control plane (for example, a multi-tenant SaaS you build on top of OpenAgno), each tenant can bring their own Meta account with their own Access Token, Phone Number ID, Verify Token, and App Secret. The external system stores these credentials encrypted (AES-256-GCM) in the `whatsapp_cloud_channels` Supabase table, and the runtime exposes a dedicated webhook per tenant.

Requires a base64-encoded 32-byte key shared between the external writer and this runtime:

```bash
# Generate once and set the SAME value on both sides
CHANNEL_SECRETS_KEY=$(node -e "console.log(require('crypto').randomBytes(32).toString('base64'))")
```

Routes (per tenant, where `tenant_id` is the `public.tenants.id` UUID):

- `GET /whatsapp-cloud/{tenant_id}/webhook` — Meta verification
- `POST /whatsapp-cloud/{tenant_id}/webhook` — inbound events

The tenant pastes this URL and the generated Verify Token in Meta Developer Console &gt; their App &gt; WhatsApp &gt; Configuration &gt; Webhook, and subscribes to the `messages` field. The runtime loads the per-tenant credentials, decrypts them with `CHANNEL_SECRETS_KEY`, validates the `X-Hub-Signature-256`, resolves the tenant's agent through the `TenantLoader`, runs the agent, and replies via Graph API with the tenant's own Access Token. Text, image, and audio messages are supported; audio uses native model audio support or OpenAI Whisper transcription when configured.

If `CHANNEL_SECRETS_KEY` is not set, the multi-tenant webhook is not mounted and the runtime still serves the single-tenant `/whatsapp/webhook` from Agno.

## Local development without a domain

During local development you usually need a public tunnel for WhatsApp, Slack, or Telegram webhooks. In OpenAgno, the practical setup is documented in [/whatsapp-tunnel](/whatsapp-tunnel).

For controlled development without a real signature:

```bash
WHATSAPP_SKIP_SIGNATURE_VALIDATION=true
```

Use that only in development.

## WhatsApp QR

```yaml
whatsapp:
  mode: qr_link
  qr_link:
    bridge_url: http://localhost:3001
```

```bash
cd bridges/whatsapp-qr
npm install
node index.js
```

```bash
SLACK_TOKEN=xoxb-your-token
SLACK_SIGNING_SECRET=your_signing_secret
```

## Telegram

```bash
TELEGRAM_TOKEN=your_botfather_token
```

## AG-UI and A2A

```bash
pip install -e '.[protocols]'
openagno add agui
openagno add a2a
```

Compatibility notes:

- `slack` and `telegram` are part of the base installation.
- `agui` and `a2a` require protocol extras.
- `ai_sdk` is not part of the supported OpenAgno runtime path.
