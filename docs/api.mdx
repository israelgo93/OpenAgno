---
title: HTTP API
description: OpenAgno runtime endpoints, including tenant provisioning, knowledge operations, admin routes, and AgentOS-native routes.
---

Typical base URL: `http://localhost:8000` (adjust host and port in deployment).

## OpenAgno custom routes

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/admin/health` | Runtime health, channels, agents, model, and scheduler state |
| `POST` | `/admin/reload` | Ask the supervisor to reload the runtime |
| `POST` | `/admin/fallback/activate` | Enable the fallback model manually |
| `POST` | `/admin/fallback/restore` | Restore the primary model |

These routes are rate-limited. If you expose them in production, protect them behind private networking or a reverse proxy.

## Knowledge routes

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/knowledge/upload` | Upload a document |
| `POST` | `/knowledge/ingest-urls` | Ingest URLs |
| `GET` | `/knowledge/list` | List indexed documents |
| `DELETE` | `/knowledge/{doc_name}` | Delete a document |
| `POST` | `/knowledge/search` | Run semantic search |

If you send `X-Tenant-ID`, these routes become tenant-aware and use Agno-native knowledge isolation.

If you define `OPENAGNO_API_KEY`, these routes require the `X-API-Key` header.

## Tenant provisioning routes

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/tenants` | List tenants |
| `POST` | `/tenants` | Create a tenant and provision its workspace |
| `GET` | `/tenants/{tenant_id}` | Read tenant metadata |
| `PATCH` | `/tenants/{tenant_id}` | Update plan, limits, slug, or workspace config |
| `DELETE` | `/tenants/{tenant_id}` | Soft-deactivate a tenant |
| `GET` | `/tenants/{tenant_id}/workspace` | Read the tenant workspace config |
| `PUT` | `/tenants/{tenant_id}/workspace` | Update the tenant workspace config |
| `GET` | `/tenants/{tenant_id}/workspace/inventory` | Read loaded main agent, sub-agents, teams, model, and WorkspaceTools state |
| `POST` | `/tenants/{tenant_id}/workspace/sub-agents` | Create or update a tenant sub-agent through WorkspaceTools |
| `POST` | `/tenants/{tenant_id}/workspace/sub-agents/{agent_id}/disable` | Disable a tenant sub-agent |
| `DELETE` | `/tenants/{tenant_id}/workspace/sub-agents/{agent_id}` | Delete a tenant sub-agent |
| `POST` | `/tenants/{tenant_id}/workspace/teams` | Create or update a tenant team through WorkspaceTools |
| `POST` | `/tenants/{tenant_id}/workspace/teams/{team_id}/disable` | Disable a tenant team |
| `DELETE` | `/tenants/{tenant_id}/workspace/teams/{team_id}` | Delete a tenant team |
| `POST` | `/tenants/{tenant_id}/reload` | Invalidate and reload the tenant runtime cache |
| `POST` | `/tenants/{tenant_id}/agents/{agent_id}/runs` | Run an agent with tenant-scoped `knowledge_filters` |

Workspace mutation routes are intended for external control planes that manage tenant workspaces over HTTP. Keep `WorkspaceTools` enabled for tenants that should self-configure sub-agents and teams, and call reload after mutations before expecting the runtime-loaded inventory to change.

## WhatsApp Cloud API

Single-tenant (operator-managed via env vars):

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/whatsapp/webhook` | Meta webhook verification |
| `POST` | `/whatsapp/webhook` | WhatsApp Cloud API events |

Multi-tenant (each tenant has its own credentials encrypted in Supabase). Requires `CHANNEL_SECRETS_KEY` to be shared between the runtime and whichever external system populates the `whatsapp_cloud_channels` table:

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/whatsapp-cloud/{tenant_id}/webhook` | Meta webhook verification for a specific tenant (`tenants.id` UUID) |
| `POST` | `/whatsapp-cloud/{tenant_id}/webhook` | Inbound events for that tenant; validates `X-Hub-Signature-256` with the tenant's App Secret and responds via Graph API with the tenant's Access Token |

## AgentOS-native routes

The runtime also exposes native AgentOS routes. The most relevant operational groups are:

| Group | Main routes |
|-------|-------------|
| Agents | `/agents`, `/agents/{agent_id}/runs` |
| Teams | `/teams`, `/teams/{team_id}/runs` |
| Sessions | `/sessions`, `/sessions/{session_id}`, `/sessions/{session_id}/runs` |
| Memories | `/memories`, `/memory_topics`, `/optimize-memories` |
| Schedules | `/schedules`, `/schedules/{schedule_id}`, `/schedules/{schedule_id}/trigger` |
| Registry / config | `/registry`, `/config`, `/components`, `/models` |
| Observability | `/health`, `/metrics`, `/traces`, `/eval-runs` |

## Example

Health check:

```bash
curl http://127.0.0.1:8000/admin/health
```

Custom knowledge search:

```bash
curl -X POST http://127.0.0.1:8000/knowledge/search \
  -H 'Content-Type: application/json' \
  -d '{"query":"OpenAgno","max_results":3}'
```

Tenant-aware run:

```bash
curl -X POST http://127.0.0.1:8000/tenants/acme/agents/agnobot-main/runs \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: change-me' \
  -d '{"message":"Summarize my tenant docs","user_id":"demo"}'
```
