## API Key

Las rutas custom de knowledge pueden protegerse con `X-API-Key`.

```bash
openssl rand -hex 32
```

En `.env`:

```bash
OPENAGNO_API_KEY=tu_key_generada
```

Si `OPENAGNO_API_KEY` no esta definida, las rutas custom de knowledge quedan abiertas en modo desarrollo.

## Rutas admin

Las rutas `/admin/*` tienen rate limiting, pero actualmente no pasan por `verify_api_key`.

En produccion:

- no las expongas sin control de red,
- protégelas con reverse proxy, firewall o autenticacion perimetral.

## WhatsApp Cloud API

### Single-tenant `POST /whatsapp/webhook`

- la firma se valida con `WHATSAPP_APP_SECRET`,
- el dedup por `message_id` ocurre solo si la firma ya paso,
- los replays completos se responden como `duplicate_ignored`,
- los payloads mixtos filtran solo los mensajes repetidos.

Solo para desarrollo local puedes usar:

```bash
WHATSAPP_SKIP_SIGNATURE_VALIDATION=true
```

### Multi-tenant `POST /whatsapp-cloud/{tenant_id}/webhook`

- Las credenciales por tenant (`access_token`, `verify_token`, `app_secret`) se cifran con **AES-256-GCM** usando una clave maestra en la variable `CHANNEL_SECRETS_KEY` (32 bytes base64). Esta clave debe ser identica entre el runtime y cualquier sistema externo que escriba las filas cifradas en Supabase, para que ambos lados puedan cifrar y descifrar.
- El cipher sigue el formato de Node `crypto.createCipheriv('aes-256-gcm')`: ciphertext + auth tag (16 bytes finales) en base64, con nonce en base64 almacenado aparte (`*_cipher` y `*_nonce`).
- Si el tenant tiene `app_secret` registrado, cada `POST` valida `X-Hub-Signature-256` con ese secreto; si no coincide el runtime responde `401 invalid_signature`.
- Los control planes hospedados deben exigir `app_secret` antes de activar trafico de clientes. El runtime conserva tolerancia para filas antiguas o desarrollo local sin `app_secret`, pero ese modo no debe usarse en produccion.
- El Verify Token tambien se cifra en Supabase; al hacer `GET /whatsapp-cloud/{tenant_id}/webhook` se descifra, se compara contra `hub.verify_token` con `hmac.compare_digest` y en exito actualiza `verified_at` en la fila del tenant.
- Los tokens nunca se escriben a disco ni a los logs del runtime; se cargan en memoria por request y se descartan al terminar la respuesta.
- Si el proveedor del modelo devuelve un error crudo, el webhook no lo reenvia literalmente por WhatsApp. El detalle queda en logs/telemetria y el usuario recibe una respuesta segura.

Para generar la clave maestra:

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

Si `CHANNEL_SECRETS_KEY` no esta definida, el runtime OSS no monta `/whatsapp-cloud/*` y registra una advertencia al arrancar.

## Rate limiting

OpenAgno aplica rate limiting a:

- rutas custom de admin,
- rutas custom de knowledge.

No se aplica rate limiting al webhook de WhatsApp Cloud API para no degradar throughput ni agravar retries de Meta.

## Inyeccion SQL

Las rutas custom de knowledge aplican una whitelist de tablas como:

- `agnobot_knowledge_contents`
- `agnobot_knowledge_vectors`

Cualquier referencia fuera de esa lista se rechaza.

## Herramientas de riesgo

Herramientas como `shell`, `python_tools` o STT/TTS deben habilitarse solo cuando el entorno y las politicas de su organizacion lo permitan.
