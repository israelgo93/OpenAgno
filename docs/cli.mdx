La ruta de operación documentada para OpenAgno es:

```bash
openagno <comando>
```

## Comandos principales

| Comando | Uso |
|---------|-----|
| `openagno init --template <id>` | Crear workspace desde template |
| `openagno templates list` | Listar templates disponibles |
| `openagno templates show <id>` | Ver metadata de un template |
| `openagno validate` | Validar `workspace/` |
| `openagno start --foreground` | Ejecutar el runtime en foreground |
| `openagno start` | Arrancar supervisor local |
| `openagno status` | Estado del runtime |
| `openagno logs --follow` | Seguir `gateway.log` |
| `openagno stop` | Detener supervisor |
| `openagno restart` | Reiniciar supervisor |

## Comandos de workspace

| Comando | Uso |
|---------|-----|
| `openagno create agent "<nombre>"` | Crear sub-agente |
| `openagno add whatsapp --mode <modo>` | Configurar WhatsApp |
| `openagno add slack` | Agregar Slack |
| `openagno add telegram` | Agregar Telegram |
| `openagno add agui` | Agregar AG-UI |
| `openagno add a2a` | Habilitar A2A |
| `openagno add tool <tool>` | Activar o desactivar tool opcional |
| `openagno deploy docker` | Levantar stack local con Docker Compose |
| `openagno deploy local` | Alias del supervisor local |

## Ejemplo completo

```bash
openagno init --template customer_support
openagno add slack
openagno add agui
openagno add tool tavily
openagno validate
openagno start --foreground
```

## Reiniciar el runtime tras cambios de configuración

Los comandos `openagno init`, `openagno add`, `openagno create agent` y cualquier edición manual de `workspace/config.yaml`, `workspace/agents/*.yaml`, `workspace/tools.yaml` o `.env` **cambian los archivos en disco pero no recargan el proceso en ejecución**. El agente ya construido queda en memoria con el modelo y los canales anteriores hasta que reinicies el runtime.

Si después de cambiar el modelo, el proveedor, las API keys o los canales los mensajes entran y devuelven `Sorry, there was an error processing your message` (o quedan en silencio), casi siempre es esta causa: el proceso sigue cargado con la configuración vieja. La señal en logs típica es ver un `ERROR ... API key not set` para el proveedor anterior aunque `.env` ya tenga la clave del proveedor nuevo.

Cómo reiniciar según tu modo de despliegue:

- **Supervisor local**: `openagno restart`
- **Foreground**: `Ctrl+C` y `openagno start --foreground` de nuevo
- **systemd**: `sudo systemctl restart openagno`
- **Docker Compose**: `docker compose restart gateway`

Una vez reiniciado, `curl http://127.0.0.1:8000/admin/health` debe reportar el `model.provider` y `model.id` nuevos, y el próximo mensaje procesarse con la configuración actualizada.

Multi-tenant: si solo cambió el workspace de un tenant específico (a través de `PUT /tenants/{tenant_id}/workspace` o editando `workspaces/<slug>/workspace/`), usa `POST /tenants/{tenant_id}/reload` para invalidar únicamente el bundle de ese tenant en el `TenantLoader`, sin reiniciar todo el proceso.
