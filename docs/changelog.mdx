## Unreleased

Trabajo acumulado en `main` desde el corte de `v1.3.0`. Estos cambios estaran
empaquetados en la proxima release.

- `openagno/channels/whatsapp_cloud.py`: nuevo router multi-tenant
  `GET/POST /whatsapp-cloud/{tenant_id}/webhook` que verifica firma
  `X-Hub-Signature-256` por tenant, descifra credenciales con
  AES-256-GCM desde `public.whatsapp_cloud_channels`, resuelve el agente
  via `TenantLoader` y responde via Graph API con el Access Token del
  propio tenant. Se monta cuando la env `CHANNEL_SECRETS_KEY` esta
  presente; si falta, el runtime sigue sirviendo el `/whatsapp/webhook`
  single-tenant de Agno y emite un log claro en el arranque.
- `bridges/whatsapp-qr/index.js`: upgrade de Baileys `6.17.16` a
  `7.0.0-rc.9` para resolver el bug `stream:error type=replaced` que
  disparaba loop infinito de QR tras el primer mensaje. Incluye
  migracion a ESM, manejo no destructivo de `loggedOut`, reintento
  inmediato en `restartRequired` (515), deteccion temprana de
  dispositivo fantasma, polling `500ms x 4` en `/send` antes de 503,
  y cola `pendingOutbox` con TTL 60s que drena al proximo `open`.
- Documentacion Mintlify alineada: nueva pagina `whatsapp-cloud-api.mdx`,
  matriz de canales con los dos modos (single-tenant y multi-tenant),
  politica de secretos AES-256-GCM en `security.mdx`.
- `AGENTS.md` al dia con el baseline actual, instrucciones de setup
  para cualquier IDE con capacidades de agente, y skills agnosticas
  bajo `.agents/skills/` (openagno, openagno-channels, mintlify).
- `CONTRIBUTING.md`, `SECURITY.md`, `.github/ISSUE_TEMPLATE/*` y
  `.github/PULL_REQUEST_TEMPLATE.md` ampliados para contribuciones
  open source estandar.

## v1.3.0

Runtime multi-tenant real con cache per-tenant, BYOK y audio por Whisper.
Ningún cambio es breaking para self-hosters single-tenant: `load_workspace()`,
la CLI y los endpoints `/tenants/*` siguen funcionando igual.

- `openagno/core/tenant_loader.py`: nuevo `TenantLoader` con cache LRU
  (variable `OPENAGNO_TENANT_CACHE_SIZE`, default 32) que resuelve el
  workspace del tenant en cada request sin reiniciar el proceso.
- `loader.load_workspace_from_dir(path)` parametrizable; `load_workspace()`
  queda como wrapper compatible.
- `_build_single_model` y `build_model` aceptan credenciales BYOK del tenant
  (`api_key`, `aws_access_key_id`, `aws_secret_access_key`) y las reenvían
  como kwargs al modelo Agno. Si no vienen, Agno sigue usando `os.environ`.
- `gateway.py`: `/whatsapp-qr/incoming` ahora exige `tenant_slug` y rutea al
  agente correspondiente via `TenantLoader`. Las rutas `/whatsapp-qr/*` se
  montan siempre, independientes del modo del workspace global.
- Audio WhatsApp: cuando el mensaje es de tipo audio y el modelo del tenant
  no soporta audio nativo, el gateway transcribe con Whisper usando la
  `api_key` del tenant (fallback a `OPENAI_API_KEY` del server) antes de
  pasar el texto al agente.
- `/admin/health?tenant_slug=X` devuelve el modelo real del tenant
  (provider, id, aws_region) sin exponer api_key ni aws_secret_*.
- `/admin/tenants/{slug}/reload` y `/tenants/{id}/reload` nuevos: invalidan
  el cache del `TenantLoader` tras cambios en el workspace.
- `bridges/whatsapp-qr/`: bridge Baileys multi-sesión con endpoints
  `/sessions/:tenantSlug/*`. Cada tenant tiene su propio directorio de
  credenciales y evento de reconexión automática tras `loggedOut`.
- `openagno/core/workspace_store.py::write_config` derrama `instructions`
  y `self_knowledge` como archivos `.md` separados en el workspace del
  tenant (el loader lee esos archivos, no solo `config.yaml`).
- `OS_SECURITY_KEY` / `JWT_VERIFICATION_KEY` se loguean al arrancar para
  dejar claro el modo de autenticación del endpoint AgentOS.
- Tests: 6 nuevos en `tests/test_byok_model.py` (forward de credenciales
  por proveedor) + 9 en `tests/test_tenant_loader.py` (cache LRU, reload,
  eviction). Suite total: 75 passing.
